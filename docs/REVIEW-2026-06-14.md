## 🏗️ Architecture & Code Quality

### Strengths
- **Clean separation of concerns** — data parsing (`lib/`), types (`types/`), UI (`components/`), and state logic (`hooks/`) are all properly separated.
- **Good use of TypeScript** — strict types on `FretValue`, `ChordDefinition`, `ChordPosition`, etc. prevent a lot of common bugs.
- **`useMemo` used correctly** in `Fretboard.tsx` for `strings` and `positionMap`, and in `App.tsx` for `positions`.
- **`useRef` for timers** — using `sessionEndRef` and `chordEndRef` with `Date.now()` instead of decrementing counters avoids timer drift. Smart.
- **`useCallback` on `start`/`stop`/`showNextChord`** — avoids unnecessary re-subscriptions in the `useEffect`.

### Issues & Suggestions

**1. `buildChordMap` uses `chord.id` but the practice levels use chord names as keys**

```ts
// usePracticeSession.ts
export function buildChordMap(chords: ChordDefinition[]): Map<string, ChordDefinition> {
  return new Map(chords.map((c) => [c.id, c]));
}
```
The `chordMap` is keyed by `c.id`, but `gyak-level-1.json` uses chord names like `"C"`, `"Am"`, `"Em"`. This only works because `id` and `name` happen to be the same value in your JSON. This implicit coupling is fragile — if IDs ever diverge from names, practice sessions will silently break. Consider keying explicitly by `name` or documenting this invariant clearly.

**2. `parseChordDatabase` hardcodes groups of 4**

```ts
for (let i = 0; i < entries.length; i += 4) {
  const slice = entries.slice(i, i + 4);
```
This silently ignores any chord that doesn't fit a multiple-of-4 grouping, and group labels are derived from index 0 and index 2. If the JSON ever has a different layout, the grouping will be wrong with no error. Consider adding a `console.warn` for unexpected group sizes, or restructuring the JSON to have explicit groups.

**3. `STRING_COUNT` is exported from `chordToPositions.ts`**

`STRING_COUNT` is a domain constant, not a utility of `chordToPositions`. It would be more appropriate in `lib/notes.ts` or a shared `lib/constants.ts` file.

**4. `chordsShown` is not reset when switching levels while stopped**

In `App.tsx`, `levelIndex` state changes call `setLevelIndex(i)`, but `chordsShown` lives inside `usePracticeSession`. Since a new hook instance gets `level` as a prop, it resets properly on `start()` — but the displayed count from a previous session lingers in the UI until the next session starts. Minor UX issue.

**5. `CHORD_TYPES` array in `App.tsx` is positionally coupled to the data**

```ts
const CHORD_TYPES = ["Dur", "Dur7", "Moll", "Moll7"] as const;
// ...
{CHORD_TYPES[i]}
```
The type label for each chord button relies on its index matching the order in the JSON. If the chord order changes, labels will be wrong. Consider adding a `type` field to `ChordDefinition` or `ChordEntry`.

## 🎨 UI & UX

### Strengths
- Dark theme is clean and well-executed with a consistent amber/orange accent.
- The fretboard is visually impressive — inlays, barre chords, open/muted string markers all work great.
- Disabling level buttons during practice (`disabled={practice.isPlaying}`) is good UX.
- Missing chord warning (`missingChords`) is a nice safeguard.

### Suggestions

**6. No visual feedback when the session ends**

When `sessionRemainingSec` hits 0, `stop()` is called silently. A toast, a brief message, or even a simple `"Vége!"` text would make the session end feel intentional rather than abrupt.

**7. `showEmptyNeck` checkbox is disabled during practice, but there's no tooltip explaining why**

Users may be confused why the checkbox is greyed out. A `title="Práktika alatt nem érhető el"` attribute on the `<label>` would help.

**8. `max-w-[125.84rem]` and `min-w-[105.6rem]` are very unusual custom values**

These seem like magic numbers. They may display poorly on very wide monitors or very narrow viewports. The inner `overflow-x-auto` on the fretboard helps, but it's worth verifying the layout on small screens (e.g. 375px wide mobile).

**9. No loading state or error boundary**

The chord database is imported statically, so it won't fail at runtime — but there's no `<ErrorBoundary>` wrapping the app. If `parseChordDatabase` throws (e.g. malformed JSON), the whole app crashes silently.

## 🔒 Robustness

**10. `pickRandomChord` retry loop is bounded but imprecise**

```ts
let attempts = 0;
while (next.id === avoidId && attempts < 8) {
  next = available[Math.floor(Math.random() * available.length)];
  attempts++;
}
```
8 attempts is reasonable, but with only 2 chords in the pool it's statistically possible to still repeat. A cleaner approach is to filter out `avoidId` from the candidate list when `available.length > 1`.

**11. `chordToPositions.ts` doesn't export `STRING_COUNT` as a named export clearly**

The `export { STRING_COUNT }` at the bottom is easy to miss. This is a minor style point — exporting it alongside the function or moving it to a constants file would be cleaner.
